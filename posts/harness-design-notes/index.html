When a research agent returns a confidently wrong answer, the instinct is to blame the model. Usually the model is fine. The failure was manufactured several steps earlier, by the harness — the retrieval, the context budgeting, the tool schemas, the retry logic — and the model merely delivered the bad news fluently.1

The claim

Most agent failures are not reasoning failures. They are context failures that arrive dressed as reasoning failures.

Stated less dramatically: if you log what actually entered the context window on each failed run, the answer was usually not retrievable from what the model was given. The model didn’t get it wrong so much as it was never given the chance to get it right.

A failure-first loop

The useful unit of work is not “improve the agent” but “characterise one failure precisely, then remove its cause.” Concretely:

  1. Freeze a corpus and a fixed set of questions.
  2. Capture every trace: retrieved chunks, the assembled prompt, tool I/O.
  3. Attribute each failure to a step, not to the agent as a whole.
  4. Fix the step. Re-run. Keep the trace.

Step 3 is where the leverage is. A flat accuracy number tells you nothing actionable; a distribution over where runs die tells you what to build next.

A minimal trace schema

from dataclasses import dataclass, field

@dataclass
class StepTrace:
    step: str                       # "retrieve" | "assemble" | "call" | "verify"
    inputs: dict
    outputs: dict
    ok: bool
    note: str = ""

@dataclass
class RunTrace:
    question_id: str
    steps: list[StepTrace] = field(default_factory=list)

    def first_failure(self) -> str | None:
        for s in self.steps:
            if not s.ok:
                return s.step      # attribute to the *earliest* broken step
        return None

Attributing to the earliest failing step matters: a wrong final answer whose real cause was an empty retrieval should be counted against retrieve, not against the model’s call.

What the numbers tend to say

A rough shape I keep seeing once failures are attributed by step:

First failing step Share of failures Typical root cause
retrieve ~45% recall gap; wrong chunking unit
assemble ~30% context truncated the key passage
call ~15% genuine reasoning error
verify ~10% no check, or the wrong check

Only the third row is the model’s fault. The other three are yours — which is good news, because they’re deterministic and you can fix them.

A small piece of the math

If a run touches \(k\) independent steps and each is reliable with probability \(p\), end-to-end reliability is \(p^{k}\). At \(p = 0.97\) and \(k = 6\) that’s about \(0.83\) — a “97% reliable” pipeline that fails one run in six. The lever isn’t a better model; it’s fewer, more reliable steps.

What to take from this

  • Instrument before you optimise. An un-traced agent is unfalsifiable.
  • Attribute failures to the earliest broken step, not the visible symptom.
  • Treat the harness as the primary surface for reliability work.
